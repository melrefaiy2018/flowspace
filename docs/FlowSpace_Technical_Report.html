1 Executive Summary

FlowSpace is a full-stack personal productivity application that consolidates Google Workspace services — Drive, Gmail, Calendar, and Tasks — into a single, AI-augmented dashboard. It delivers real-time workspace intelligence through an embedded conversational agent capable of reading from and writing to all connected Google services via a tool-calling loop.

The system ships in three deployment forms:

  • Web application — Express + Vite on localhost:3000
  • macOS desktop application — Tauri v2 with bundled Node.js runtime
  • Docker container — Node 22 Alpine, single-command deployment

2 System Architecture

FlowSpace follows a two-layer monolith pattern: a single Express.js server serves both the REST/SSE API and the frontend assets. In development, Vite's dev server is mounted as Express middleware for hot module replacement. In production, pre-built static files are served directly.

3.2 In-Memory Cache

All read endpoints use a 60-second TTL in-memory cache to minimize Google API calls. Cache keys are scoped per endpoint. The briefing endpoint uses a longer 10-minute TTL as its content changes less frequently and requires an LLM call to generate.

3.3 Run Tracking

Each chat interaction spawns a "run" tracked in an in-memory Map<string, RunRecord> with 24-hour auto-pruning. Runs transition through states: queued → running → awaiting_approval | completed | failed. The frontend's RunCenter component visualizes this pipeline.

4 AI Agent System

The AI agent is the core differentiator of FlowSpace. It converts natural language instructions into concrete actions across Google Workspace, using a multi-round tool-calling loop.

4.2 Chat Loop Architecture

User sends message
  │
  ▼

  │
  ▼
┌─── Tool-Calling Loop (max 5 rounds) ─────────────────────────┐
│  1. Send messages + system prompt to LLM                      │
│  2. LLM responds with tool_calls[] or final text              │
│  3. For each tool_call:                                       │
│     ├── isWriteTool? → buildApprovalRequest() → pause for UI  │
│     └── Read tool? → executeTool() → append result            │
│  4. Extract memories from tool results                        │
│  5. Append tool results to message history                    │
│  6. Loop back to step 1                                       │
└───────────────────────────────────────────────────────────────┘
  │
  ▼
Emit assistant_complete with structured blocks + suggestions

4.3 System Prompt

The system prompt (~160 lines) is dynamically constructed per request. It includes: the current date/time in the user's timezone, persistent thread context (brief), retrieved long-term memories, tool usage guidelines, and output formatting rules (structured JSON blocks, 2-3 follow-up suggestions).

4.5 Tool Execution

All Google Workspace operations are executed through the gws CLI as a subprocess (execFile, not shell). The access token is passed via the GOOGLE_WORKSPACE_CLI_TOKEN environment variable — never as a command-line argument. Each tool call has a 30-second timeout and supports abort signals.

6 Desktop Application (Tauri v2)

FlowSpace ships as a native macOS application using Tauri v2, a Rust-based framework that provides a lightweight alternative to Electron (~200 MB vs ~500 MB+).

6.1 Architecture

The Tauri shell (lib.rs, 112 lines) spawns the Express server as a child process, waits for it to become ready on port 3000, then navigates its WebView to http://localhost:3000. This approach allows the same codebase to run in both browser and desktop modes.

7 Authentication & Multi-Account

7.1 OAuth Flow

FlowSpace delegates OAuth entirely to the gws CLI (Google Workspace CLI), eliminating the need for users to create a Google Cloud project or configure OAuth client secrets.

User clicks "Sign in with Google" │ ▼ [1] GET /api/auth/gws-status │── gws not installed? → POST /api/auth/gws-install │── gws not authenticated? → continue ▼ [2] POST /api/auth/gws-login │── Spawns: gws auth login -s drive,gmail,calendar,tasks,userinfo │── Opens browser → Google consent screen │── User authorizes → gws stores token ▼ [3] POST /api/auth/gws-import │── Spawns: gws auth export --unmasked │── Parses JSON credentials │── Saves to .gws-credentials.{accountKey}.json │── Upserts account record in .accounts.json ▼ [4] GET /api/auth/status → { authenticated: true, user: {...} }

7.2 Credential Hierarchy

When resolving credentials, the system tries sources in order:

  1. Active account — .accounts.json → active account's credential file
  2. Legacy credentials — .gws-credentials.json (single-account fallback)
  3. OAuth2 web flow — .tokens.json
  4. Application Default Credentials — ~/.config/gcloud/application_default_credentials.json (dev-only)

7.3 Multi-Account Support

FlowSpace supports multiple Google accounts. Each account is stored as an entry in .accounts.json with its own credential file. Users can switch accounts via the sidebar account menu. All data (cache, conversations, preferences) is scoped per account.

9 Tool System & Approval Workflow

9.1 Approval Flow for Write Operations

All write tools (sending emails, creating events, modifying spreadsheets, etc.) require explicit user approval before execution. This prevents the AI from taking unintended actions on the user's behalf. 

LLM returns tool_call for write operation (e.g., send_email)
  │
  ▼
buildApprovalRequest(toolName, args)
  │── Creates ApprovalRequest with:
  │   • title, summary
  │   • editable fields (to, subject, body, ...)
  │   • before/after preview (when applicable)
  │
  ▼
Emit ChatStreamEvent { type: 'tool_event', status: 'approval_required' }
  │── Run status → 'awaiting_approval'
  │── Frontend shows approval card in ChatThread
  │
  ▼
User reviews and clicks "Approve" or "Dismiss"
  │
  ├── Approve → POST /api/chat/runs/:runId/approve
  │              → executeTool() with approved args
  │              → emit result, continue chat loop
  │
  └── Dismiss → dismissApproval()
               → tool_call marked as rejected
               → LLM informed, may suggest alternative

9.2 Dynamic Tool Composition

Users can create custom tools by composing existing tools into multi-step workflows. These are stored in .dynamic-tools.json and loaded into the tool registry at runtime. 

// Example: Custom "expense tracker" tool
{
  "name": "log_expense",
  "description": "Log an expense to the team spreadsheet",
  "steps": [
    { "action": "sheets_read", "args": { "spreadsheet_id": "abc123" }, "outputKey": "sheet" },
    { "action": "sheets_append", "args": { "spreadsheet_id": "abc123", "values": "{{input}}" } }
  ]
}

10 Memory & Personalization

10.1 Long-Term Memory

The AI agent maintains a long-term memory system that persists across conversations. Memories are stored as JSON files in .memory/{userHash}.json within DATA_DIR.

ComponentFileResponsibility
Memory Storememory-store.tsCRUD operations on memory entries
Memory Retrievermemory-retriever.tsRetrieve relevant memories for context injection
Memory Extractormemory-extractor.tsExtract memorable facts from tool results

10.2 Persona System

Users can customize the AI agent's personality and behavior through the Persona Settings. The persona prompt is injected into the system prompt, influencing tone, formality, and response style. Personas are stored per-account in .persona.{accountKey}.json.

10.3 Importance Feedback

The briefing system includes preference learning: users can mark items as more or less important, and these preferences influence future briefing generation. Feedback is stored via /api/briefing/save-preference.

13 Security Model

13.1 Authentication Security

  • OAuth2 flow delegated to Google via gws CLI (no custom OAuth server)
  • Credentials stored in user's home directory with filesystem permissions
  • Access tokens refreshed automatically; refresh tokens never exposed to frontend
  • Token passed to gws CLI via environment variable, never command-line argument

13.2 Data Privacy

  • Conversations: Stored in browser localStorage (never sent to a server)
  • Memories: Stored in user's local filesystem (DATA_DIR)
  • No telemetry: No analytics, tracking, or data collection
  • No cloud backend: All processing happens locally (except LLM API calls)

13.3 Write Protection

  • All write operations require explicit user approval via the approval workflow
  • Bulk inbox actions show preview of affected items before execution
  • Undo capability for inbox actions via audit log
  • Max 5 tool-calling rounds prevents runaway execution

13.4 Code-Level Safeguards

  • No eval() or dynamic code execution
  • Subprocess execution via execFile (not exec/shell) prevents injection
  • Error messages sanitized before sending to client (no stack traces)
  • CSP headers configured in Tauri for desktop builds
  • .gitignore excludes all credential and token files

15 Design Decisions & Trade-offs

DecisionRationaleTrade-off
Express monolith Single process serves API + frontend. Simpler deployment, shared auth context, no CORS. Cannot scale API and frontend independently.
gws CLI for auth Zero GCP setup for users. No project creation, no client secrets to manage. Runtime dependency on gws CLI binary.
In-memory cache No database required. Fast dashboard loads. Simple implementation. Data is eventually consistent (60s stale). Lost on server restart.
SSE for chat Standard HTTP streaming. No WebSocket complexity. Works through proxies. Unidirectional — approval responses require separate POST.
Tauri over Electron ~200 MB vs ~500 MB+. Native performance. Rust for reliability. macOS-only currently. Smaller ecosystem than Electron.
localStorage for chat Private by default. No cloud dependency. Immediate persistence. Not synced across devices. Storage limited (~5-10 MB).
Tool approval flow Prevents AI from taking unintended actions. User remains in control. Adds friction to write operations. Cannot fully automate workflows.
GLM-4.7 default Cost-effective. Good tool-calling support. OpenAI-compatible API. Less capable than GPT-4/Claude for complex reasoning. Configurable as mitigation.
