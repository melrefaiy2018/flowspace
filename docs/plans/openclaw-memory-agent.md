## Problem Statement

FlowSpace has a powerful reactive AI agent — it does deep work when you ask. But it never acts without being prompted. The goal is a proactive layer: the agent learns what matters to you over time, and uses that knowledge to do the work before you ask. You open the app in the morning and find a draft ready, not a blank inbox.

**The business problem:** Nobody except the founder has used FlowSpace yet. The proactive memory loop needs user signals to learn from. Without users, the compounding loop has nothing to compound. The original plan built floor 2 (proactive agent) before anyone walked through floor 1 (reactive dashboard).

**Revised thesis:** Distribution and habit formation are the immediate wall. Memory is the long-term moat. Ship something useful on day one to get users in the door, build the memory layer in parallel.

## Target User

**Who:** Executives, founders, and senior ICs who can't afford (or don't have) a human chief of staff or EA. Solo startup founders, VPs at Series A companies, heads of product with 15+ meetings/week.

**What gets them fired:** Missing a board follow-up. Showing up to a client call without context. Dropping an investor email thread. Forgetting a commitment they made in a sent email 3 days ago.

**Current workaround:** 30 minutes of manual prep before every important meeting — scanning email, searching Drive, re-reading Slack threads. Or worse: showing up cold.

## Revised Strategy: Two Phases

### Phase 1: Ship Day-One Value (weeks 1-2)

**Goal:** Get real users using FlowSpace's reactive features. Validate that people connect Google accounts, use the dashboard, and come back for day 2.

**What ships:**
- Meeting Prep Draft Queue — NO memory required. Daily cron (node-cron, no OpenClaw dependency) scans calendar 24-72h ahead, generates briefs for meetings with external attendees, surfaces them in a Draft Queue panel.
- Simple "was this useful?" boolean toggle per draft (not the full importance model — just collects data for Phase 2).
- Works out of the box: `npx flowspace` → sign in → see tomorrow's meetings prepped.

**Why this first:** Fyxer charges $28/mo for meeting notes. Reclaim charges $10/mo for calendar scheduling. Meeting prep that works on day one is a product people already pay for. If nobody uses this, the memory layer doesn't matter.

**Success metric:** 10+ users connect Google accounts and open the Draft Queue at least twice in the first week.

### Phase 2: Memory + OpenClaw Proactive Layer (weeks 3-6)

**Goal:** Layer the importance model on top of validated usage. OpenClaw integration for background scheduling. Memory starts compounding from Phase 1 approval/dismissal signals.

**What ships:**
- ImportanceSignal memory category (bounded weights, 30-day decay)
- OpenClaw integration for scheduled background scans
- Confidence scoring that evolves with learned preferences
- The full proactive loop: scan → prep → queue → approve/dismiss → learn → repeat

**Gate:** Only proceed to Phase 2 if Phase 1 validates. If nobody uses the Draft Queue in Phase 1, the memory layer is premature.

## Constraints

- Must work with the existing approval gate (no autonomous writes)
- Phase 1: no OpenClaw dependency. `node-cron` or system crontab is sufficient.
- Phase 2: OpenClaw as infrastructure layer for background scheduling
- "Read-only" applies to the horizon scanner session only. Post-approval execution uses the existing write-capable approval flow in `chat.ts`.
- Cold start (Phase 1): simple heuristic, no memory signals needed
- Cold start (Phase 2): importance model bootstraps from Phase 1 data

## Premises (Revised)

1. **Distribution is the immediate wall.** The memory model can't train without users. Phase 1 must deliver value on day one without memory.
2. **Memory is the long-term moat.** Once users are in, the compounding importance model is what makes FlowSpace irreplaceable. Open-source + self-hosted is the durable moat.
3. OpenClaw is **infrastructure** (not competition, not distribution). FlowSpace registers agents, passes credentials, receives output. OpenClaw handles scheduling.
4. The meeting prep use case is **independently valuable** — people pay $10-28/mo for this today. It doesn't need the full memory loop to be worth using.
5. Background sessions are sandboxed read-only. All writes gated behind user approval.
6. The OpenClaw API contract must be defined before Phase 2 integration begins.

## Architecture

### Phase 1 Architecture (No OpenClaw)

```
System cron or node-cron (6am daily)
  -> flowspace horizon-scan (CLI command)
       - calendar_agenda(48h window)
       - For each meeting (>= 30 min, >= 2 external attendees):
           - search_emails(attendee emails, last 7 days)
           - search_drive(meeting title, attendee names)
           - Generate brief (LLM)
       -> POST /api/drafts/ingest (Express server writes staged-drafts.json)

User opens app:
  -> Draft Queue panel
       - View brief
       - "Useful" / "Not useful" toggle (simple boolean, stored per draft)
       - Approve => triggers execution
       - Dismiss => records as not useful
```

### Phase 2 Architecture (OpenClaw + Memory)

```
OpenClaw Cron (configurable schedule)
  -> Horizon Scanner Agent (read-only, restricted toolset)
       - Same scan as Phase 1, BUT:
       - Confidence scoring informed by ImportanceSignal weights
       - Skip meetings with entities below weight threshold (-3.0)
       - Increased context depth for high-weight entities
       -> POST /api/drafts/ingest

User opens app:
  -> Draft Queue panel (enhanced)
       - Confidence badge with tooltip showing reasoning
       - Inline edit before approve
       - Approve/dismiss/edit signals feed ImportanceSignal store
```

## Technical Design

### New Components

**1. `src/agent/horizon-scanner.ts`** (new file, ~200 lines)

Phase 1: standalone Node script, called via `flowspace horizon-scan` CLI command.
Phase 2: called by OpenClaw as a shell command.

Imports a **restricted** toolset from `tools.ts` — whitelist of read-only tools only
(`calendar_agenda`, `search_drive`, `search_emails`, `docs_read`). Enforced at import
time, not via a flag. (Note: codebase uses `search_emails`, not `search_emails`.)

**Auth bootstrapping:** The scanner reads credentials from `DATA_DIR/.gws-credentials.{email}.json`
directly (same files the server uses). No running server required for auth. The CLI
command resolves `DATA_DIR` the same way `server.ts` does.

Output: JSON to stdout. Caller POSTs to `/api/drafts/ingest` if server is running.
If server is down (system crontab at 6am), the CLI wrapper writes directly to
`DATA_DIR/staged-drafts.json` using atomic write (write to temp file, rename).
This is the one exception to "scanner never writes" — only when server is unreachable.

**`StagedDraft` type:**

```typescript
interface StagedDraft {
  id: string;               // uuid
  meetingId: string;        // Google Calendar event ID
  meetingTitle: string;
  meetingTime: string;      // ISO 8601
  attendees: string[];      // email addresses
  summary: string;          // LLM-generated brief (markdown, max 500 words)
  linkedDocs: { title: string; url: string }[];
  relatedEmails: { subject: string; from: string; snippet: string }[];
  suggestedActions: string[];
  confidence: number;       // Phase 1: always 1.0. Phase 2: 0.0-1.0 heuristic.
  confidenceReason: string; // Phase 1: "". Phase 2: e.g. "2 prior meetings approved"
  useful?: boolean;         // Phase 1: simple toggle. Phase 2: replaced by signals.
  createdAt: string;        // ISO 8601
  seenAt?: string;          // set when GET /api/drafts is called
  status: "pending" | "approved" | "dismissed" | "executed" | "error";
}
```

**Cold-start heuristic (Phase 1, zero signals):**
Prep meetings >= 30 min with >= 2 external attendees (not @same-domain).
"External" = attendee email domain differs from the authenticated user's primary email domain.

**Batch limits:** Max 10 meetings per scan to stay within Google API quotas and control LLM cost.
Per-meeting LLM failure: skip that meeting, log warning, continue with remaining meetings.

**Draft retention:** Auto-purge drafts older than 7 days or whose `meetingTime` has passed.

**Catch-up scan:** If the server starts and the last scan was >12h ago, run a scan immediately
(handles cases where the cron fired while the server was down).

**Confidence scoring (Phase 2):**
- Base: 0.4 for meetings >= 30 min with >= 2 attendees
- +0.3 if attendees have positive importance signals (prior approvals)
- +0.2 if Drive search returns >= 1 relevant file
- +0.1 if related emails found in last 7 days
- Cap at 1.0

---

**2. Memory extension: `ImportanceSignal` category** (Phase 2 only)

**`ImportanceSignal` is a separate data structure** stored in its own file
(`DATA_DIR/.importance-signals.json`), not a new `MemoryCategory` variant.
This avoids breaking existing memory queries that switch on category type.
`memory-store.ts` gets a new exported function `getImportanceWeight(entityId)` 
that the scanner calls during Phase 2 confidence scoring.

```typescript
// src/agent/memory/importance-signals.ts (new file)
interface ImportanceSignal {
  entityId: string;         // attendee email | meeting type keyword | domain
  signal: "approved" | "dismissed" | "edited" | "ignored";
  weight: number;           // bounded: -10.0 to +10.0
  context: string;          // what was prepped
  updatedAt: string;        // ISO 8601
}
```

- Weight **decay**: halved every 30 days
- **Skip threshold**: weight < -3.0 → entity excluded from future scans
- **"Seen but ignored"**: `GET /api/drafts` sets `seenAt`. Pending 48h after seenAt → signal = "ignored", delta = -0.2
- **"Edit" detection**: `POST /api/drafts/:id/approve` accepts optional `editedContent`. Different from original → signal = "edited", delta = +0.7
- All writes through Express — scanner never writes directly

**Bootstrap from Phase 1:** The `useful` boolean from Phase 1 drafts maps to Phase 2 signals: `useful=true` → weight +1.0 per attendee; `useful=false` → weight -0.5. This means Day 1 of Phase 2 already has data.

**3. `/api/drafts` endpoints** (in `server.ts`)

| Method | Path | Behavior | Phase |
|--------|------|----------|-------|
| POST | `/api/drafts/ingest` | Scanner output → staged-drafts.json. Localhost-only (bound to 127.0.0.1). | 1 |
| GET | `/api/drafts` | Returns drafts; sets `seenAt` on pending | 1 |
| POST | `/api/drafts/:id/approve` | Approve + execute. Optional `editedContent`. | 1 |
| POST | `/api/drafts/:id/dismiss` | Dismiss. Records negative signal (Phase 2). | 1 |
| PATCH | `/api/drafts/:id/useful` | Toggle `useful` boolean. | 1 |

**5. Scheduling**

Phase 1 (no OpenClaw):
- `node-cron` in-process: schedules `horizon-scan` when FlowSpace server is running
- OR system crontab: `0 6 * * * flowspace horizon-scan`
- Failure: scanner outputs `{ status: "error", message: "..." }`, UI shows "Last scan failed"

Phase 2 (OpenClaw):
- OpenClaw registers `flowspace horizon-scan` as a daily trigger
- Richer scheduling (configurable per user, multiple times/day)
- Lifecycle management: register, update, deregister agents

## Business Model

### Pricing (to validate)

**Open-core model** (most likely path):
- **Free tier:** Self-hosted, full reactive dashboard, BYO LLM, manual meeting prep
- **Pro tier ($15-25/mo):** Proactive scanning, Draft Queue, importance memory, priority support
- **Team tier ($TBD):** Multi-user, shared importance context, admin controls

**Unit economics estimate:**
- LLM inference per meeting brief: ~$0.02-0.05 (few thousand tokens)
- 10 meetings/week × $0.04 = ~$1.60/month per user
- At $20/mo, gross margin is >90%

**Validation approach:** Ship Phase 1 as fully free. Add a "this would be a paid feature" label on confidence scoring and importance learning (Phase 2 previews). Measure interest.

## Cross-Model Perspective

**Codex cold read:** The strongest version is a proactive Chief of Staff that closes the loop between detection, preparation, and preference learning. The most revealing answer was "full loop — nothing smaller is worth paying for" — this reveals a risky philosophy of optimizing for end-state ambition over early validation. Codex's challenged premise: the wall is distribution and habit formation, not memory. 48-hour prototype recommendation: meeting prep draft queue with node-cron, simple boolean useful toggle, skip the full memory model.

**Synthesis:** Both Claude and Codex independently concluded that the "full loop or nothing" stance needed revision. The refined plan splits into two phases: ship day-one value without memory (Phase 1), then layer memory on top of validated usage (Phase 2). The `useful` boolean from Phase 1 becomes the bootstrap data for Phase 2's importance model.
